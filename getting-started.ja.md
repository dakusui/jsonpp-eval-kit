## 1. 前提ツール

以下の 3 つが `PATH` に必要です:

| ツール | 確認 | 無い場合 |
|---|---|---|
| Go 1.24+ | `go version` | https://go.dev/dl/ |
| `jq` | `jq --version` | `brew install jq` / `apt install jq` |
| `ruby` | `ruby --version` | macOS はプリインストール済み。Linux は `apt install ruby` |

## 2. ツールの入手（`jq++` と `yq++`）

```sh
git clone https://github.com/dakusui/jqplusplus.git
cd jqplusplus
go build -o bin/jq++ ./cmd/jqplusplus          # bin/jq++ をビルド
export PATH="$PWD/bin:$PWD/tools/bin:$PATH"     # jq++ (bin), yq++ (tools/bin)
```

> **すでに `jq++` をインストール済みの場合も**、上記の手順はそのまま実行してください。
> `yq++` は `jq++` ではなく `jqplusplus` リポジトリに同梱される別のスクリプトなので、
> 以前の `jq++` インストールには含まれません。この手順で `yq++` が PATH に通り、この
> 課題で検証済みの `jq++`（v0.0.38 以降）が入ります。`export PATH` の行により、この
> build が既存の古い `jq++` より優先されます。

確認:

```sh
jq++ --version                # 例: jq++ version v0.0.38
command -v yq++               # PATH にあれば yq++ のパスを表示
```

- **`yq++`** は `.yaml++` ソースを YAML に変換します — この課題で使うのはこちらです。
- **`jq++`** はその内部エンジンで、検証（§4）では直接呼び出します。[^tools]

## 3. JSON++ を 5 分で

JSON++ は、普通の YAML/JSON にいくつかの予約キーを足したものです。必要なのは次のとおり:

| 構文 | 意味 |
|---|---|
| `$extends: [base.yaml++]` | 1 つ以上のベースファイルを継承。後のフィールドが前を上書きし、現在のファイルが親より優先されます。 |
| *（オブジェクトのフィールド）* | **深くマージ** — 子オブジェクトが親オブジェクトにキー単位でマージされます。 |
| *（配列のフィールド）* | **マージされず置換** — 子が配列フィールドを指定すると、その配列が親の配列を丸ごと置き換えます。 |
| リスト要素への `$extends` | 配列の 1 要素（例: 1 つのステップ）だけベースを継承し、異なるフィールドだけ上書きできます。 |
| `eval:string:EXPR` | 継承の解決 *後* に評価される `jq` 式（例: `refexpr(".foo")` は別フィールドを参照）。 |
| `_name:`（アンダースコア接頭辞） | 式から利用するための非公開「ホルダー」フィールド。**出力からは取り除かれます**。[^holder] |

> **さらに詳しく:** JSON++ / jq++ の詳細なリファレンスは
> <https://dakusui.github.io/jqplusplus/> にあります。この課題に必須ではありませんが、
> 気になったとき、または AI アシスタントに手伝ってもらう場合（AI が詳細ドキュメントを
> 参照できます）に便利です。

### 小さな実例

テキストファイルを 2 つ作成し（JSON++ ソースは `.yaml++` 拡張子を使います）、
2 つ目に対して `yq++` を実行します。

`greeting.yaml++`:
```yaml
greeting: Hello
```

`hello.yaml++`:
```yaml
$extends: [greeting.yaml++]
name: Mark
```

実行:
```sh
yq++ hello.yaml++
```
```yaml
greeting: Hello
name: Mark
```

子は `greeting` を継承し、`name` を追加しました。要はこれだけです — 繰り返される
部分をベースに切り出し、各ファイルには差分だけを残します。

## 4. 課題の進め方

1. JSON++ ソース（`.yaml++` ファイル）を `work/starter/`（課題 1 用）に **作成** します: 共通する構造をベースファイルに
   置き、各リーフがそれを `$extends` して、差分だけを残します。`starter-corpus/` は
   変更しないでください — これと照合して検証します。
2. `$extends` がベースを見つけられるよう、**`jq++` に共有ベースの場所を教えます**:
   ```sh
   export JF_PATH=path/to/your/shared
   ```
3. リーフから YAML を **生成** します:
   ```sh
   yq++ your-leaf.yaml++
   ```
4. 出力が元と一致することを **検証** します（意味的な一致 — キーの順序や書式は問いません）:
   ```sh
   diff <(jq++ starter-corpus/ci-api.yml | jq -S .) \
        <(jq++ your-leaf.yaml++     | jq -S .) && echo "MATCH"
   ```
   意味を変えてしまうリファクタリングは失敗です — 生成された出力は、元と同じデータを
   持っているべきです。

## 5. 知っておくとよいこと

- **キーはソートされて出力され**、書式は正規化されます — これは想定どおりです。同一性は
  バイト列ではなく *データ* についてのものです。
- **YAML のコメントは** ラウンドトリップで保持されません。
- **`on:`** は出力で `'on':` になることがあります — これは正しい挙動です（`on` を
  YAML のブール値 `true` ではなく文字列として保つため）。GitHub Actions はこれを
  受け付けます。
